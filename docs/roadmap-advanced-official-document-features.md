# 后续功能路线图：正式公文增强能力

本文记录 `official-docx-formatter` 在“可信格式引擎”重构完成后，可以继续加入的正式公文增强能力。

当前重构的第一目标是：先把普通正文型公文的识别、诊断、格式计划、执行和报告做稳。下面这些能力很有价值，但复杂度更高，建议作为后续独立阶段加入。

---

## 1. 页码模块

### 要解决什么

正式公文的页码不是简单在页脚写一个数字。它通常涉及：

- 页码位置。
- 页码字号和字体。
- 奇偶页页码是否放在外侧。
- 是否保留已有页脚内容。
- 是否使用 Word 的 `PAGE` 域，而不是写死数字。

### 为什么要单独做

如果直接在页脚里写数字，文档页数变化后页码可能不更新。更稳的做法是插入 Word 字段，例如当前页 `PAGE`，必要时再配合总页数 `NUMPAGES`。

### 第一版建议能力

- 检测是否已有页码。
- 默认不覆盖非页码页脚内容。
- 支持插入普通页码。
- 支持奇偶页外侧页码。
- 在 report 中说明：
  - 是否发现已有页码。
  - 是否替换。
  - 是否保留原页脚内容。

### 暂不做

- 复杂分节页码。
- 封面不编号。
- 目录页单独编号。
- 多种公文模板页脚。

---

## 2. 表格内部格式整理

### 要解决什么

国企、机关公文里经常有附件表、整改清单、责任分工表、台账。正文格式整理好了，但表格内部可能仍然很乱：

- 字体不统一。
- 单元格边距不统一。
- 行距不统一。
- 表格宽度超过版心。
- 边框乱。
- 列宽不合理。

### 为什么要单独做

表格不是普通段落。直接套正文规则可能会把表格弄坏，例如：

- 内容被挤成很多行。
- 列宽变化导致版面溢出。
- 原来有意义的边框被覆盖。
- 合并单元格处理出错。

### 第一版建议能力

- 只处理表格内部字体、字号、行距。
- 表格宽度限制在页面版心内。
- 保守设置单元格边距。
- 默认保留已有合并单元格。
- 默认保留已有表头结构。
- report 中说明处理了几个表格、跳过了哪些复杂表格。

### 暂不做

- 自动理解表格语义。
- 自动重建复杂台账。
- 自动拆分跨页表格。
- 自动生成统计表。

## 3. 红头、版记、印章、页脚域

这一组听起来比较“公文术语”，可以拆开理解。

### 3.1 红头

红头就是公文首页上方的红色发文机关标识和红色分隔线，例如：

```text
某某单位文件
```

下面通常有一条红色横线，再往下是标题、主送机关、正文。

#### 难点

- 不同单位红头样式差异很大。
- 字体、字号、红线位置、发文字号位置都可能有单位模板。
- 仅靠普通 DOCX 段落很难保证和正式模板完全一致。

#### 建议做法

红头不要靠规则硬画，优先支持“单位 Word 模板”：

```text
template.docx + 正文内容 + profile
```

也就是说，工具负责把正文放进模板指定位置，而不是凭空生成一个单位红头。

### 3.2 版记

版记是公文末页底部的印发机关、印发日期等信息区域。它常见于正式印发件的最后一页底部，例如：

```text
某某办公室                        2026年6月21日印发
```

#### 难点

- 它通常不是正文的一部分。
- 位置接近页面底部。
- 需要和正文、附件、页码共存。
- 如果正文长度变化，版记位置也要稳定。

#### 建议做法

第一阶段只支持：

- 检测是否已有版记。
- 对已有版记做字体和位置整理。
- 不自动凭空新增版记，除非用户提供模板或明确字段。

### 3.3 印章

印章指盖章位置，不只是插入一张图片。正式公文里印章通常要覆盖落款和日期的一部分，位置有规范要求。

#### 难点

- 印章图片涉及单位真实印章，安全性很高。
- 盖章位置和落款日期关系复杂。
- Word、WPS、PDF 的最终渲染可能有差异。
- 自动插入真实印章容易引发误用风险。

#### 建议做法

不要默认自动加真实印章。

可以做两个较安全的能力：

- 检查落款日期区域是否给印章留够空间。
- 在用户提供测试章或占位章时，生成“盖章位置预览稿”。

正式印章应仍由单位流程或人工确认。

### 3.4 页脚域

页脚域指 Word 里的动态字段，例如：

- `PAGE`：当前页码。
- `NUMPAGES`：总页数。
- 日期字段。
- 文档属性字段。

它们不是普通文字，而是 Word 会自动更新的字段。

#### 为什么重要

如果写死页码，文档改动后页码可能错。使用字段后，Word/WPS 打开或更新域时能重新计算。

#### 建议做法

页码模块里优先使用字段，而不是写死页码数字。

## 4. 标准目录生成

### 要解决什么

有些长文档、制度汇编、报告材料需要目录。目录不是简单复制标题，而是要和正文标题层级绑定。

### 可能能力

- 根据标题层级识别目录项。
- 生成 Word 标准目录字段。
- 支持一级、二级、三级标题。
- 目录页码使用 Word 自动域。
- report 中说明识别了哪些标题作为目录项。

### 关键前提

必须先稳定识别标题层级：

```text
一、一级标题
（一）二级标题
1. 三级标题
（1）四级标题
```

并且要避免把正文中的编号句误判成目录标题。

### 第一版建议能力

- 对长文档提供可选参数：

```bash
--generate-toc
```

- 默认不生成目录。
- 只在识别出足够清晰的标题层级时生成。
- 如果标题层级混乱，只报告“无法可靠生成目录”，不强行生成。

### 暂不做

- 多级复杂目录样式。
- 图表目录。
- 附件目录。
- 自动改写标题以适配目录。

---

## 建议实施顺序

建议在可信引擎重构之后，按下面顺序加：

1. 页码模块。
2. 表格内部格式整理。
3. 标准目录生成。
4. 红头、版记、印章、页脚域。

原因：

- 页码和目录都依赖 Word 字段，但页码更小，可以先建立字段处理经验。
- 表格在国企材料中高频，实用价值很高。
- 目录依赖稳定标题层级，适合在结构识别成熟后加。
- 红头、版记、印章最接近正式印发模板，最好等 profile/template 体系更成熟后再做。

---

## 总体原则

- 每个增强能力都必须有诊断模式。
- 每个增强能力都必须写入 report。
- 默认保守，不确定就不改。
- 优先支持用户提供的单位模板，而不是凭空猜正式模板。
- 不把这些能力混进正文润色或内容生成。
